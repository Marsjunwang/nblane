# nblane · 初步设计闭环

本文档从当前仓库已存在的产品定义、设计手册与架构边界出发，补齐一个
可执行的初步闭环：**用户为什么来、系统如何协作、结果写回哪里、如何验证
这个循环真的成立。**

## 1. 这份闭环文档解决什么问题

当前文档体系已经分别回答了三个问题：

- [product.md](product.md) 回答“为什么做、最终想成为什么”。
- [design.md](design.md) 回答“先做哪些里程碑、命令和契约怎么落地”。
- [architecture.md](architecture.md) 回答“仓库今天真正实现到了哪里”。

还缺的一环是：**把这三者串成一个可以跑起来的最小产品闭环。**

这里的“闭环”不是一句抽象口号，而是下面六件事连续成立：

1. 用户带着真实任务进入系统。
2. 系统能基于已有 prior 判断当前能做什么、缺什么。
3. Human 与 Agent 能围绕该任务协作推进。
4. 结果会写回个人或团队资产，而不是停留在聊天里。
5. 下一次任务到来时，系统能利用上一次沉淀的内容。
6. 团队和产品负责人能判断这条链路是否真的在变好。

## 2. 当前版本的最小闭环定义

在 `v0.2` 阶段，nblane 真正应该跑通的不是“全自动 Agent 公司”，而是下面
这条 **Human + Agent + Team 的最小工作回路**：

```text
任务进入
  → 读取个人 prior
  → 判断能力与缺口
  → Human + Agent 结对推进
  → 结果写回个人 / 团队资产
  → 重新生成 context
  → 下一任务复用更新后的 prior
```

把这条链路翻成当前仓库能力，就是：

| 阶段 | 产品动作 | 当前实现 |
|------|----------|----------|
| 任务进入 | 用户提出一个真实任务 | `nblane gap <name> "<task>"` |
| 读取 prior | 读取 `SKILL.md`、`skill-tree.yaml`、可选 `agent-profile.yaml` | `context` / `gap` / `status` |
| 判断缺口 | 判断能做什么、缺什么、先补哪里 | `tools/gap.py` |
| 协作推进 | Human 与 Agent 结对执行 | 通过 `context` 输出系统提示，外部 Agent 执行 |
| 结果写回 | 更新成长记录、技能状态、团队共享池 | 目前以手工写文件为主 |
| 复用下一次 | 重新输出系统提示，进入下一任务 | `nblane context <name>` |

这就是当前版本最重要的设计边界：**闭环已经存在，但“回写”和“团队复用”
仍是半自动。**

## 3. 从产品视角看，闭环的核心价值

### 3.1 对个人

nblane 不是帮用户“做一次任务”，而是让每次任务都反过来校准用户的长期
能力模型。任务只是入口，长期可复用的 prior 才是资产。

### 3.2 对 Agent

Agent 不再只接收一个临时 prompt，而是接收一个会持续更新的个人操作系统。
这使 Agent 从“即时回答器”变成“长期协作接口”。

### 3.3 对团队

团队的价值不在于多人同时在线，而在于把问题、项目、证据和方法持续沉淀到
共享产品池。否则每个人都在和 Agent 单打独斗，无法形成组织复利。

### 3.4 对产品

真正的产品护城河不是某个命令，而是下面这件事能否越来越自然：

```text
任务完成一次
  → 系统更懂人
  → 团队多一份共享资产
  → 下一次同类任务更快、更准、更少重复劳动
```

## 4. 从设计视角看，用户路径应该怎样闭合

### 4.1 主路径：个人任务闭环

这是当前最应该打磨的主路径，也是最接近已实现能力的一条线。

```text
用户提出任务
  → 系统判断任务对应哪些技能节点
  → 系统给出可做部分 / 缺口 / 下一步建议
  → Human 与 Agent 一起交付
  → 用户更新 SKILL.md / skill-tree / log / kanban
  → 重新生成 context，进入下一轮
```

这条路径必须满足三个设计要求：

- **进入成本低**：一句任务描述即可进入，不要求先填复杂表单。
- **反馈可解释**：为什么匹配这个节点、为什么判断有缺口，用户能看懂。
- **回写负担可接受**：写回动作不能重到让用户不愿维护。

### 4.2 次路径：团队协作闭环

当多个成员都有自己的个人闭环后，系统再向团队层扩展：

```text
个人任务命中团队问题
  → 写入 / 对齐 product pool
  → 团队看见重复问题与可复用资产
  → 协作分工
  → 结果回写 pool
```

当前仓库已经有 `teams/<team_id>/team.yaml` 与 `product-pool.yaml` 作为数据落点，
但智能写回与自动路由还没有真正产品化。

### 4.3 第三路径：公开展示闭环

这条路径还未实现，但设计上已经明确：

```text
个人 / 团队资产沉淀
  → Agent 提炼可公开内容
  → Human 审核
  → 生成个人页 / 团队页 / 分享卡片
```

因此它不应被当作独立功能，而应被视为个人闭环与团队闭环的自然输出层。

## 5. 从架构视角看，闭环必须依赖哪些对象

### 5.1 个人层对象

| 对象 | 作用 | 当前状态 |
|------|------|----------|
| `profiles/{name}/SKILL.md` | 人类 prior、长期方向、成长记录 | 已实现 |
| `profiles/{name}/skill-tree.yaml` | 结构化技能状态 | 已实现 |
| `profiles/{name}/kanban.md` | 当前任务面 | 已实现 |
| `profiles/{name}/agent-profile.yaml` | Agent 对用户理解的结构化补充 | 已实现，可选 |

### 5.2 团队层对象

| 对象 | 作用 | 当前状态 |
|------|------|----------|
| `teams/{id}/team.yaml` | 团队元数据、成员、规则 | 已实现 |
| `teams/{id}/product-pool.yaml` | 问题/项目/证据/方法/决策池 | 已实现 |

### 5.3 运行时能力

| 命令 | 在闭环中的角色 |
|------|----------------|
| `init` | 初始化一个可进入闭环的 profile |
| `context` | 物化当前 prior，交给 Agent 使用 |
| `status` | 查看当前技能状态 |
| `validate` | 保证结构化数据可信 |
| `gap` | 触发“任务 -> 能力判断 -> 缺口建议” |
| `team` | 查看团队共享池概况 |
| `log` | 写入显式成长事件 |

一个关键架构判断是：**nblane 的产品核心在数据对象，不在服务端。**
所以闭环是否成立，本质上取决于这些文件之间能否持续形成稳定的数据回路。

## 6. 当前版本真正闭合到哪一步

如果从“能跑起来”而不是“愿景完整”来看，当前版本已经闭合了前半段：

```text
任务输入
  → 基于 schema 与 skill-tree 做规则判断
  → 输出 gap 与 next step
  → 通过 context 把 prior 交给 Agent
```

还没有完全闭合的，是后半段：

```text
交付结果
  → 自动更新个人状态
  → 自动同步团队共享池
  → 自动生成公共展示
```

所以更准确的判断不是“产品还没闭环”，而是：

- **个人判断闭环** 已具备基础形态。
- **个人回写闭环** 目前依赖人工维护。
- **团队复用闭环** 已有数据壳，缺自动化。
- **公开展示闭环** 仍在路线图。

## 7. 把“闭环”定义成产品验收标准

如果要让这个产品从文档走向真正可验证，建议把闭环验收改写成下面五类指标。

### 7.1 任务进入是否顺畅

- 用户是否能用一句自然语言任务直接触发分析。
- `gap` 输出是否足够清晰，不需要先懂 schema 内部实现。

### 7.2 判断是否可信

- 用户是否认同“匹配到的技能节点大体合理”。
- 用户是否能看懂缺口为什么出现。
- `validate` 是否能先兜住脏数据问题。

### 7.3 回写是否真的发生

- 一次任务结束后，用户是否愿意更新 `SKILL.md`、`skill-tree.yaml` 或 `log`。
- 团队是否愿意把结果写入 `product-pool.yaml`。

### 7.4 下一次是否复用成功

- 下一次相似任务中，系统是否能更快给出更准的判断。
- 团队是否减少重复研究与重复沟通。

### 7.5 是否形成外部可见成果

- 是否能从内部资产稳定提炼出项目、证据、方法与公开叙事。
- 人类是否仍牢牢控制公开边界。

## 8. 建议的 Phase 0 产品闭环

如果只做一个“当前就能跑”的版本，建议把产品定义收敛为下面这条 Phase 0
闭环：

```text
1. 维护个人 profile
2. 用 gap 分析真实任务
3. 用 context 驱动 Agent 协作
4. 任务结束后手工回写个人资产
5. 如涉及团队，手工回写 shared product pool
6. 下一次继续用更新后的 prior 工作
```

这条闭环足够小，但已经满足产品成立的必要条件：

- 有真实输入，而不是空谈成长。
- 有结构化判断，而不是纯主观感受。
- 有执行协作，而不是只写文档。
- 有资产沉淀，而不是一次性聊天。
- 有下一轮复用，而不是每次从零开始。

## 9. 下一步最值得补的三块

为了把“初步闭环”升级成“稳定闭环”，优先级建议如下：

### 9.1 先补个人回写半自动化

目标不是一开始就全自动，而是把任务结束后的回写模板标准化，例如：

- 本次任务对应哪些技能节点
- 哪些节点状态应更新
- 哪条证据应写入项目或日志
- 哪些内容值得进入 `agent-profile.yaml`

### 9.2 再补团队池同步

让团队层至少能回答三件事：

- 这项任务属于哪个团队问题
- 结果应沉淀为证据、方法还是决策
- 谁最适合继续拥有这个任务

### 9.3 最后补公开面导出

公开页不应先做成“好看页面”，而应先做成“从内部结构化资产可稳定导出”。
否则 public surface 会沦为又一套手工维护内容。

---

## 10. 一句话结论

以当前产品、设计与架构状态来看，nblane 的初步设计闭环已经可以明确为：

> **让真实任务驱动 Human + Agent 协作，把能力判断与任务结果写回个人和团队
> 资产，并在下一轮任务中复用这些沉淀。**

当前版本最重要的工作，不是继续扩概念，而是把这条闭环从“可理解”变成
“可反复发生”。
